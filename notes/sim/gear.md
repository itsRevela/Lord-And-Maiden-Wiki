## EntryEffect decode (gear & accessories ‚Äî `data/sim/gear.json` `equipment`/`accessories`/`set_bonuses`)

`PropInfo.Effect` for type 3 is one or more `effectId_value` tokens joined by `+`, decoded via
`EntryEffect.csv` exactly as the game's `GetEntryDes(ToEntryList(...))`:

- **`DataType` 1 = flat** ‚Üí `+value` (e.g. `10_15` ‚Üí Soldier ATK +15).
- **`DataType` 2 = percent** ‚Üí `+(value / Size * 100)%`; every percent attr has `Size = 10000`
  (e.g. `30_3000` ‚Üí Soldier ATK +30%; `46_1050` ‚Üí PVE DMG Dealt Increased +10.5%).

Each gear `PropInfo` row is a **fully-built tier** ‚Äî there is no per-row leveling, so the row's
`Effect` IS the maxed value for that tier (`PropInfo.Value` = the piece's Power rating, not a level).
The catalog therefore lists every tier ‚òÖ1‚Üí‚òÖ6; the simulator picks the tier the hero wears.

EntryEffect id groups used by gear: 1‚Äì5 HP flat, 6‚Äì10 ATK flat, 11‚Äì15 DEF flat, 16‚Äì20 DES flat,
21‚Äì25 HP %, 26‚Äì30 ATK %, 31‚Äì35 DEF %, 36‚Äì40 DES %, 41‚Äì45 March Spd %, 46/47 PVE/PVP DMG Dealt %,
48/49 PVE/PVP DMG Taken Reduced %, 50 Hero ATK Spd (flat), 51/52/53/54 Hero ATK/DEF/DES/Soldiers
(flat), 131/132 Equip Tactical/Pursuit Skill Activation Probability %.

### Set bonuses (`set_bonuses`)

From the gear `timeinfo` field, format `setId$threePieceEffect+sixPieceEffect`
(`GetPropDes` l.8767‚Äì8788). A piece counts toward a set if its `timeinfo` `setId` matches; equipping
**‚â•3** same-set pieces grants `threePieceEffect`, **‚â•6** grants `sixPieceEffect`. Both effect strings
decode through the same EntryEffect rules. 35 set-tier rows (Hunting‚Ä¶Fate across ‚òÖ2‚Äì‚òÖ6) ‚Äî matches
`wiki/Items/Equipment.md` exactly. Accessories (slots 7/8), relics, runes have **no** set bonus.

## Relics (`hero_relics`) ‚Äî PropInfo type 5

- **Attachment:** a relic's `PropInfo.PosType` is the **owning hero's id** ‚Äî a relic equips only in
  that one hero's Relic slot (verified: relic 19001 PosType 1 = HeroInfo[1] "Crolos"). 98 hero relics
  (68 ‚òÖ5 + 30 ‚òÖ4), matching `wiki/Heroes/Relics.md`.
- **What it influences:** `gotoPanel = "ST_Num"` names the **Talent skill** the relic enhances
  (e.g. Crolos relic ‚Üí skill `3_12`, Patra relic ‚Üí Tactical `2_81` "Ghost Bone"). The relic boosts
  that skill ‚Äî higher trigger/effect probability, a damage/heal coefficient, real-damage base, or a
  flat attribute swing ‚Äî so a relic is a **combat proc/coefficient modifier**, not a raw stat stick.
- **Levels & max:** rows are Lv1..max (`PropInfo.Value` = level); ‚òÖ4 relics ‚Üí Lv4, ‚òÖ5 ‚Üí Lv5.
  `gear.json` stores the **max-level** row. In-game **Score** = `rarity * level * 1800`
  (maxed ‚òÖ5 = 45,000; maxed ‚òÖ4 = 28,800), per `GetPropDes` l.8800.
- **Effect namespace:** the relic `Effect` numeric code is a **skill-buff id** (the `Buff.csv`
  namespace ‚Äî e.g. `6` = DMG Dealt Reduced, `8` = DMG Taken Reduced, `1` = Effect/Skill Trigger
  Probability, `10` = DEF Attribute), **NOT** EntryEffect. So `gear.json` keeps the raw
  `effect_tokens_max` and uses the client's `des_en` as the authoritative readable `max_bonus`.
  Only one relic (Patra, `2_0.30+41_7.2`) has a multi-token Effect.
- **Server-side:** duplicate-relic count consumed per level ‚Üí `UNKNOWN_SERVER_SIDE`.

## Runes (`runes`) ‚Äî PropInfo type 9

- **One rune slot per hero** (PosType 10). 105 rune kinds ‚Äî one per boostable Tactical/Pursuit skill,
  matching `wiki/Heroes/Runes.md`. No set bonus.
- **Boosted skill:** `gotoPanel = "ST_Num"` (e.g. `2_22`); resolved to the skill name + type and the
  skill's own `Des_en` (what the boosted skill does).
- **Effect = `45_<frac>`.** Per the client's `des_en` and the task rule, **EffectType 45 means
  "Skill Trigger Probability"** in the rune context (NOT EntryEffect 45 = "Soldier March Spd", which
  is its meaning on gear). `trigger_chance_pct = frac * 100`. ‚òÖ4 runes ‚Üí Lv4, ‚òÖ5 ‚Üí Lv5;
  `gear.json` stores Lv1 and **max** trigger chance (e.g. Brutal Chase 1.6%‚Üí8%, Mushin 2%‚Üí10%).
- **Server-side:** Runes-Fragment cost per level, dismantle yield, box rates ‚Üí `UNKNOWN_SERVER_SIDE`.

## Skill Awaken (`skill_awake`) ‚Äî SkillAwake.csv

- **What it is:** the per-skill **Awaken** leveling track (the 4th hero-growth track from
  `Hero-Advancement.md`). Keyed by `(skill_type, skill_id)`; 187 skills, ST dist 1/2/3/4 =
  Strategic/Tactical/Passive/Pursuit. Each skill has 4 or 5 awaken levels.
- **Awaken vs the Skill-Stone item:** `SkillAwake` is the **leveling table for a skill**, not the
  skill-stone *item*. A Skill Stone is a separate chest item (absent from the static files, per
  `wiki/Heroes/Skill-Stones.md`) that *grants an extra skill* to a hero; once granted, that skill is
  awakened through this same `SkillAwake` track ‚Äî "the granted skill's awaken is shared with the
  hero's own skill track" (`Skill-Stones.md` l.18). So: SkillAwake = skill leveling; Skill Stone =
  the item that adds the skill. Evidence: `SkillAwake.csv` has no item id, only `(skill_type,
  skill_id, Lv, Effecs)`.
- **`Effecs` is CUMULATIVE per level** (the value at that level, not an increment): e.g. skill
  `1_12` Lv1 `10_3.2` (DEF +3.2) ‚Üí Lv4 `10_12.8` (DEF +12.8). `gear.json` stores the **max-Lv** row.
  The task's example `10_3.2` is the **Lv1** value; the MAX cumulative is `10_12.8`.
- **Namespace:** `Effecs` uses the **buff-id namespace** (same as relics / `Buff.csv`), NOT
  EntryEffect ‚Äî `10` = DEF Attribute, `3` = Healing Coefficient, `1` = Effect Trigger Probability.
  `des_en` is the authoritative readable; `gear.json` keeps `max_buff_id`/`max_buff_value` + `max_bonus`.
